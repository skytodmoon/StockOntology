# StockOntology 数据库职责与前后端交互设计

## 一、数据库职责划分

### 1. 数据库职责矩阵

| 数据库类型 | 数据库名称 | 端口 | 业务职责 | 数据类型 | 典型查询场景 |
|-----------|-----------|------|----------|----------|-------------|
| **图数据库** | Neo4j | 7687 | 知识图谱存储、供应链关系、概念关联、龙头战法分析 | 节点、关系、属性 | 供应链查询、关联分析、路径搜索 |
| **关系型数据库** | PostgreSQL | 5432 | 结构化数据存储、企业基本信息、行业分类、用户管理 | 结构化表格 | CRUD操作、复杂查询、报表统计 |
| **时序数据库** | TimescaleDB | 5433 | 股票行情数据、时序数据分析、K线数据、实时数据流 | 时间序列数据 | 时序查询、聚合分析、趋势预测 |
| **缓存/消息队列** | Redis | 6379 | 热点数据缓存、Session管理、任务队列、实时数据推送 | Key-Value、List、Pub/Sub | 缓存加速、实时推送、任务调度 |

### 2. 各数据库详细职责

#### 📊 Neo4j - 图数据库（核心业务）

**职责定位**：知识图谱核心存储，管理实体间复杂关系

**存储内容**：
- **节点**：Company（企业）、Industry（行业）、Concept（概念）、MarketEvent（市场事件）、Investor（投资者）
- **关系**：BELONGS_TO（所属）、SUPPLIES（供应）、COMPETES_WITH（竞争）、IMPACTS（影响）、BELONGS_TO_CONCEPT（概念归属）
- **属性**：企业市值、护城河等级、行业分类、概念热度等

**核心查询场景**：
- 供应链上下游分析：`MATCH (c)-[:SUPPLIES*1..3]->(target) RETURN path`
- 龙头股关联分析：`MATCH (leader:Company{isLeader:true})-[]-(related) RETURN related`
- 概念板块扩散：`MATCH (c:Concept)-[]-(companies) RETURN companies`

---

#### 🗄️ PostgreSQL - 关系型数据库（结构化数据）

**职责定位**：结构化业务数据存储，支持复杂事务

**存储内容**：
- `companies`：企业基本信息（代码、名称、行业、员工数等）
- `industries`：行业分类层级
- `investors`：投资者信息
- `users`：系统用户管理
- `system_config`：系统配置项

**核心查询场景**：
- 企业列表分页查询
- 多条件筛选统计
- 数据报表生成

---

#### ⏰ TimescaleDB - 时序数据库（行情数据）

**职责定位**：股票行情时序数据存储与分析

**存储内容**：
- `stock_prices`：股票日K线（开盘价、收盘价、最高价、最低价、成交量）
- `stock_minutes`：股票分钟级数据（高频交易数据）
- `market_index`：大盘指数数据

**核心查询场景**：
- 历史行情查询：`SELECT * FROM stock_prices WHERE stock_code = '600519' AND time > '2024-01-01'`
- 时序聚合：`SELECT time_bucket('1 day', time), avg(close) FROM stock_prices GROUP BY 1`
- 趋势分析：移动平均线、MACD等技术指标计算

---

#### 🚀 Redis - 缓存与消息队列

**职责定位**：性能加速、实时通信、任务调度

**存储内容**：
- **缓存层**：热点股票数据、用户Session、配置信息
- **消息队列**：实时行情推送、异步任务队列
- **计数器**：访问频率限制、页面访问量

**核心使用场景**：
- 热点数据缓存：`GET stock:600519:latest_price`
- 实时推送：Pub/Sub模式推送行情更新
- 任务队列：Celery异步任务

---

#### 🔍 ChromaDB - 向量数据库（语义检索）

**职责定位**：文档语义检索、RAG知识库

**存储内容**：
- 企业研究报告嵌入
- 行业分析文档向量
- 新闻资讯语义向量
- LLM对话上下文

**核心查询场景**：
- 语义搜索：`query("半导体行业分析")` → 返回相关文档
- RAG增强：为LLM提供上下文知识
- 文档推荐：基于相似度的内容推荐

### 3. 核心 API 设计

#### 3.1 Graph API - 知识图谱查询

| API路径 | HTTP方法 | 功能 | 参数 | 返回值 |
|---------|----------|------|------|--------|
| `/api/v1/graph/company/{stockCode}/graph` | GET | 获取企业关联图谱 | `depth`: 深度(默认2) | nodes, edges |
| `/api/v1/graph/supply_chain/{stockCode}` | GET | 获取供应链关系 | `direction`: up/down/all | 供应链节点列表 |
| `/api/v1/graph/concept/{conceptCode}/stocks` | GET | 获取概念成分股 | 无 | 股票列表 |
| `/api/v1/graph/query` | POST | 自定义Cypher查询 | `query`: Cypher语句, `params`: 参数 | 查询结果 |

#### 3.2 Company API - 企业数据

| API路径 | HTTP方法 | 功能 | 参数 | 返回值 |
|---------|----------|------|------|--------|
| `/api/v1/companies/` | GET | 获取企业列表 | `page`, `size`, `industry`, `isLeader` | 分页企业列表 |
| `/api/v1/companies/{stockCode}` | GET | 获取企业详情 | 无 | 企业完整信息 |
| `/api/v1/companies/{stockCode}/financial` | GET | 获取财务数据 | 无 | 财务指标 |
| `/api/v1/companies/{stockCode}/dragon` | GET | 获取龙头分析 | 无 | 龙头评分、护城河分析 |

#### 3.3 Market API - 行情数据

| API路径 | HTTP方法 | 功能 | 参数 | 返回值 |
|---------|----------|------|------|--------|
| `/api/v1/market/prices/{stockCode}` | GET | 获取历史行情 | `start_date`, `end_date`, `interval` | K线数据 |
| `/api/v1/market/real_time/{stockCode}` | GET | 获取实时行情 | 无 | 实时价格 |
| `/api/v1/market/index/{indexCode}` | GET | 获取大盘指数 | `start_date`, `end_date` | 指数数据 |
| `/api/v1/market/analysis/{stockCode}` | GET | 行情分析 | `period`: day/week/month | 技术指标 |

#### 3.4 LLM API - 智能问答

| API路径 | HTTP方法 | 功能 | 参数 | 返回值 |
|---------|----------|------|------|--------|
| `/api/v1/llm/chat` | POST | 智能问答 | `question`: 问题, `context`: 上下文 | 回答结果 |

## 四、关键设计原则

### 1. 数据一致性策略

- **最终一致性**：主数据写入PostgreSQL，异步同步到Neo4j
- **缓存策略**：Redis作为热点数据缓存，设置合理过期时间
- **事务边界**：单个数据库内保证ACID，跨库操作通过消息队列保证最终一致

### 2. 性能优化策略

- **读写分离**：PostgreSQL/TimescaleDB配置读写分离
- **缓存分层**：Redis本地缓存 + CDN静态资源缓存
- **索引优化**：各数据库根据查询模式创建合适索引
- **异步处理**：非实时任务通过Celery异步执行

### 3. 扩展性设计

- **数据库抽象层**：Repository模式封装数据库访问
- **服务注册发现**：支持水平扩展
- **消息队列解耦**：通过Redis/Celery实现异步解耦

## 五、安全与监控

### 1. 安全设计

- **数据库访问控制**：最小权限原则，各服务使用独立账号
- **数据加密**：传输加密(TLS)、敏感数据存储加密
- **SQL注入防护**：使用参数化查询，禁止拼接SQL
- **API认证**：JWT Token认证，支持OAuth2

### 2. 监控与告警

- **数据库监控**：Prometheus + Grafana监控各数据库指标
- **慢查询日志**：记录并分析慢查询
- **健康检查**：定期检查数据库连接状态
- **告警机制**：异常情况自动告警

## 六、总结

### 数据库职责总结

| 数据库 | 核心价值 | 典型用例 |
|--------|----------|----------|
| **Neo4j** | 关系图谱 | 供应链分析、龙头战法、概念扩散 |
| **PostgreSQL** | 事务可靠 | 企业数据管理、用户系统 |
| **TimescaleDB** | 时序分析 | K线数据、行情分析、趋势预测 |
| **Redis** | 性能加速 | 缓存、实时推送、任务队列 |
| **ChromaDB** | 语义检索 | RAG问答、文档推荐 |

### 架构优势

1. **职责清晰**：各数据库专注于自身擅长的领域
2. **性能优化**：通过缓存和异步处理提升响应速度
3. **扩展性强**：各组件可独立扩展，支持高并发
4. **数据一致性**：通过合理的同步策略保证数据质量

此设计为StockOntology项目提供了清晰的数据库职责划分和前后端交互架构，支持龙头战法分析的核心业务需求。
